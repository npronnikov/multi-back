# Todo Text Editing Specification Delta

## Purpose

Эта delta-spec добавляет возможность редактирования текста существующих todo-сообщений. Пользователи могут исправлять опечатки, обновлять информацию или изменять содержимое сообщений после их создания без удаления и пересоздания.

## Requirements

### ADDED Requirements

#### Requirement: UI для редактирования текста
Система должна предоставлять UI-механизм для редактирования текста существующего todo.

##### Scenario: Открытие режима редактирования
- **WHEN** Пользователь кликает на кнопку "Edit" рядом с todo-сообщением
- **THEN** Открывается modal dialog с текущим текстом todo в поле ввода
- **AND** Кнопки "Cancel" и "Save" доступны
- **AND** Фокус устанавливается на поле ввода

##### Scenario: Отмена редактирования
- **WHEN** Пользователь кликает на кнопку "Cancel" в modal dialog
- **THEN** Modal dialog закрывается без сохранения изменений
- **AND** Текст todo остаётся без изменений
- **AND** UI возвращается в исходное состояние

##### Scenario: Сохранение с валидным текстом
- **WHEN** Пользователь вводит валидный текст (непустой, не превышает лимит) и кликает "Save"
- **THEN** Modal dialog закрывается
- **AND** Optimistic update: UI показывает обновлённый текст сразу
- **AND** Отправляется PUT запрос к `/api/todos/{id}` с `{ text, completed }`
- **AND** При успешном ответе сервера UI остаётся в обновлённом состоянии
- **AND** При ошибке сервера UI откатывается к исходному тексту и показывается сообщение об ошибке

---

#### Requirement: Валидация ввода при редактировании
Система должна валидировать ввод текста при редактировании todo.

##### Scenario: Попытка сохранить пустой текст
- **WHEN** Пользователь удаляет весь текст и кликает "Save"
- **THEN** Кнопка "Save" отключена или показывает ошибку валидации
- **AND** Modal dialog не закрывается
- **AND** Показывается сообщение "Text is required"

##### Scenario: Попытка сохранить слишком длинный текст
- **WHEN** Пользователь вводит текст превышающий лимит (например, > 500 символов)
- **THEN** Показывается счётчик символов с предупреждением
- **AND** Кнопка "Save" отключена или показывает ошибку валидации
- **AND** Modal dialog не закрывается

##### Scenario: Ввод валидного текста
- **WHEN** Пользователь вводит текст в допустимых пределах (1-500 символов)
- **THEN** Поле ввода показывает текст без ошибок
- **AND** Кнопка "Save" активна

---

#### Requirement: Сохранение изменений через API
Система должна сохранять изменения текста через существующий PUT endpoint.

##### Scenario: Успешное сохранение
- **WHEN** Пользователь кликает "Save" с валидным текстом
- **THEN** Отправляется PUT запрос к `/api/todos/{id}` с body:
  ```json
  {
    "text": "новый текст todo",
    "completed": false
  }
  ```
- **AND** При получении успешного ответа (HTTP 200) UI подтверждает обновление
- **AND** Optimistic update остается в силе

##### Scenario: Ошибка сервера при сохранении
- **WHEN** PUT запрос возвращает ошибку (HTTP 4xx/5xx)
- **THEN** UI откатывает текст к исходному значению
- **AND** Показывается сообщение об ошибке: "Failed to save changes"
- **AND** Modal dialog закрывается
- **AND** Todo остаётся в исходном состоянии

##### Scenario: Потеря соединения
- **WHEN** PUT запрос не удаётся из-за отсутствия сети
- **THEN** UI откатывает текст к исходному значению
- **AND** Показывается сообщение: "Network error. Please check your connection."
- **AND** Modal dialog закрывается

---

#### Requirement: Обновление UI после редактирования
Система должна корректно обновлять UI после успешного редактирования.

##### Scenario: Отображение отредактированного текста
- **WHEN** Редактирование успешно завершено
- **THEN** Todo-сообщение показывает новый текст в списке
- **AND** Положение todo в списке не меняется
- **AND** Статус completed не изменяется
- **AND** Кнопки "Edit" и "Delete" доступны для этого todo

##### Scenario: Сохранение фокуса после редактирования
- **WHEN** Пользователь сохраняет изменения и закрывает modal
- **THEN** Фокус возвращается на кнопку "Edit" отредактированного todo
- **AND** Поддерживается keyboard navigation

---

### REMOVED Requirements

Отсутствуют. Это новая capability без удалений.
