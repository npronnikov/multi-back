# Design: Add Version Button

## Overview
Добавляем кнопку "Version" в UI Todo List приложения и создаем backend endpoint `GET /api/version`. При клике на кнопку frontend делает запрос к backend и отображает полученную версию через alert/popup. Это простое и практичное решение, которое использует существующую архитектуру HTTP-общения между frontend и backend.

## Decisions

### Decision: Использовать alert() для отображения версии
**Выбор:** Показывать версию через `window.alert()` с простым текстовым сообщением.

**Обоснование:**
- Минимальное количество кода
- Не требует дополнительных UI компонентов (modal, popover)
- Удобно для MVP и быстрой проверки
- Соответствует простоте текущего Todo List UI

**Альтернативы:**
- Использовать custom modal/dialog — больше кода, но лучше UX
- Показывать inline текст на странице — требует перерисовки UI и место для размещения

### Decision: Endpoint возвращает JSON с version и timestamp
**Выбор:** Backend endpoint возвращает `{ "version": "string", "timestamp": "string" }`.

**Обоснование:**
- Структурированный формат легко парсить
- Timestamp добавляет контекст (когда была собрана версия)
- Расширяемый формат в будущем можно дополнить полями (git commit, environment)

**Альтернативы:**
- Возвращать только version string — менее информативно
- Возвращать plain text — сложнее парсить и расширять

### Decision: Расположение кнопки в header
**Выбор:** Разместить кнопку "Version" в header рядом с заголовком "Todo List".

**Обоснование:**
- Логично: информация о версии — глобальная для приложения
- Не смешивается с основным контентом (todo items)
- Header уже существует, нужно только добавить компонент

**Альтернативы:**
- В footer внизу страницы — меньше заметно, требует добавления footer
- Отдельной секцией — избыточно для одного элемента

### Decision: Spring Boot @Value для получения версии
**Выбор:** Использовать `@Value("${project.version}")` для чтения версии из pom.xml.

**Обоснование:**
- Стандартный подход для Spring Boot приложений
- Версия уже задана в pom.xml
- Не требует hardcoded значений

**Альтернативы:**
- Hardcode версию в коде — плохо для поддержки
- Чтение из MANIFEST.MF — сложнее и менее надежно

### Decision: Кэширование версии в React state
**Выбор:** Сохранять полученную версию в state компонента и использовать кэш при повторных кликах.

**Обоснование:**
- Снижает нагрузку на backend (не дергать endpoint на каждый клик)
- Быстрый отклик для пользователя
- Версия редко меняется за сессию

**Альтернативы:**
- Делать запрос на каждый клик — избыточно
- Не хранить в state — сложнее управлять состоянием загрузки

## Implementation Notes

### Backend (Spring Boot)
1. Создать `VersionController` в пакете `com.example.demo.controller`
2. Использовать `@Value("${project.version}")` для инъекции версии из pom.xml
3. Создать DTO record `VersionDTO` с полями `version` и `timestamp`
4. Endpoint должен быть доступен по пути `/api/version`
5. Использовать `Instant.now().toString()` для timestamp в ISO-8601

### Frontend (Next.js + React)
1. Создать компонент `VersionButton` в `app/components/`
2. Использовать `useState` для хранения версии, состояния загрузки и ошибок
3. Использовать `fetch` для запроса к `http://localhost:8081/api/version`
4. При клике проверять кэш: если версия уже загружена — показывать сразу, иначе — запрос
5. Использовать `window.alert()` для отображения версии
6. Во время загрузки отключать кнопку (disabled attribute)

### Интеграция с существующим кодом
- Frontend уже использует fetch для взаимодействия с backend (см. `app/page.tsx`)
- Backend порт: 8081 (из существующего кода)
- Frontend порт: 8082
- Использовать существующий паттерн обработки ошибок

## Risks

### Риск: Версия не инжектится из pom.xml
**Митигация:** Убедиться, что в pom.xml задан `${project.version}` и Spring Boot корректно резолвит это свойство. При необходимости добавить resource filtering в build конфигурацию.

### Риск: CORS ошибка при запросе с frontend
**Митигация:** Backend уже должен иметь CORS конфигурацию для работы с frontend (существующий код это подтверждает). Проверить, что `/api/version` включен в разрешенные endpoints.

### Риск: Alert блокирует UI
**Митигация:** Для MVP это приемлемо. В будущем можно заменить на non-blocking modal или toast notification.

### Риск: Кэширование версии может показать устаревшие данные
**Митигация:** Для текущего использования это не критично. В будущем можно добавить кнопку "Refresh" или периодическую инвалидацию кэша.
